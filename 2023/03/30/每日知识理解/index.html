反射

什么是反射机制？说说反射机制的优缺点、应用场景

Java 反射机制就是允许程序在运行时动态的获取类的信息，创建对象以及调用方法的机制。

Java 的反射机制提供了运行时检测 Java 类型信息的能力，让Java 可以通过程序来获取其本身的信息

反射的优点

  • 可以动态地获取信息，不需要在编译时就获取类的信息

  • 可以动态地创建对象，不需要在编译的时候就直到对象的类型

  • 可以动态地调用对象的属性和方法，可以在运行的时候动态地改变对象的行为

缺点

  • 要通过反射来获取到对象的信息，不如直接调用对象的方式来得快，因此效率较低
  • 会破坏类本身的封装性，可能让代码变得复杂和不稳定

应用场景

  • 动态代理，动态代理可以在运行时动态地创建代理对象，不需要在编译时就获取到接口的实现类:question:
  • 单元测试，Junit 等单元测试使用反射的机制动态地获取类和方法信息，实现自动化测试 :question:
  • 加载配置文件，许多框架 Spring 使用反射机制来读取解析配置文件，实现 DI 和 AOP 功能 :question:

动态代理

理解

动态代理是由我们传入一个指定的对象和它实现的接口，获取到一个代理对象，我们调用任何代理对象的方法，最后都会转发到我们实现的中介类里面，在中介类里面通过反射调用，同时我们也可以在中介类中间写上我们自己的处理逻辑，比如计算方法运行时间，打印方法的日志…等等。

为什么我们会需要动态代理(动态代理的应用场景)

动态代理的优点(相比于静态代理)

  • 优点在于我们可以很方便的对代理方法进行统一的处理，

即我们可以只修改中间的中介类，就可以让所有实现了这个中介类的类做出相应的改变，而不用一个一个做修改(静态代理)，同时还具有业务类和代理类解耦的优点(静态代理)

代理模式详解

代理模式：给某一个对象提供一个代理，并由代理对象来控制对真实对象的访问。代理模式是一种结构型设计模式。

代理模式角色分为 3 种：

Subject（抽象主题角色）：定义代理类和真实主题的公共对外方法，也是代理类代理真实主题的方法；

RealSubject（真实主题角色）：真正实现业务逻辑的类；

Proxy（代理主题角色）：用来代理和封装真实主题；

代理模式的结构比较简单，其核心是代理类，为了让客户端能够一致性地对待真实对象和代理对象，在代理模式中引入了抽象层

如果根据字节码的创建时机来分类，可以分为静态代理和动态代理：

  • 所谓静态也就是在程序运行前就已经存在代理类的字节码文件，代理类和真实主题角色的关系在运行前就确定了。
  • 而动态代理的源码是在程序运行期间由JVM根据反射等机制动态的生成，所以在运行前并不存在代理类的字节码文件

ThreadLocal

ThreadLocal 可以理解为一个是每个线程存储当前线程值的一个集合，ThreadLocal 内部维护着一个 ThreadLocalMap 这个 Map 存储 Entry 对象，Entry 键是 一个 ThreadLocal 对象，值是对应传入的泛型，也就是存储的值，当我们调用 ThreadLocal.get() 方法的时候，实际上就是从 ThreadLocalMap 中取出当前线程对应的变量，从而得到线程隔离的目的

Jvm垃圾回收 —> JVM学习

如何排除full Gc

什么是GC

在JVM领域中GC（Garbage Collection）翻译为 “垃圾收集“，Garbage Collector翻译为 “垃圾收集器”。

分代模型

新生代

Eden是内存中的一个区域， 用来分配新创建的对象。通常会有多个线程同时创建多个对象，所以Eden区被划分为多个线程本地分配缓冲区(Thread Local Allocation Buffer, 简称TLAB)。通过这种缓冲区划分，大部分对象直接由JVM 在对应线程的TLAB中分配, 避免与其他线程的同步操作。需要注意的是，这里的对象是分配在对应的线程分配缓冲区当中的。有点类似于TreadLocal

当有新的对象创建的时候，JVM会先将对象分配给线程单独的缓冲区，若没有空间，则会分配给共享内存区域，如果还没有空间，就会进行一次针对新生代的GC，回收后若还没有空间，则会将对象分配给老年代

存活区(Survivor Spaces)

Eden区的旁边是两个存活区, 称为 from 空间和 to 空间。需要着重强调的的是, 任意时刻总有一个存活区是空的(empty)。

空的那个存活区用于存放当年轻代 GC 的时候收集的对象，包括另一个非空的存活区的对象也会放入这个存活区。这个时候，存活区两者的状态就会发生互换

空(from)—–>非空。 非空(to)—->空

这就意味着二者的状态取决于新生代的 GC ?

为什么要这么设计 , 对象的转移应该伴随着CPU占用，为什么要增加这个操作呢？

存活的对象会在两个存活区之间复制多次，直到某些对象的存活时间达到一定的阀值。分代理论假设, 存活超过一定时间的对象很可能会继续存活更长时间。

这类“ 年老” 的对象因此被提升(promoted )到老年代。提升的时候， 存活区的对象不再是复制到另一个存活区,而是迁移到老年代, 并在老年代一直驻留, 直到变为不可达对象。

此外GC会跟踪记录每个存活区对象存活的次数，每次分代GC完成后，存活对象的年龄就会+1。当年龄超过提升阈值(tenuring threshold)（18？），就会被提升到老年代区域。

peace:

这里大概就能回答前面的那个问题，JVM每次进行GC的时候都需要对存活区的对象进行判定，如果存活区的对象存活超过一定时间 (在多次GC中存活下来，就会被转移到老年区中去)，那如何判定这个一定时间呢？

GC 针对这个对存活区的对象分配了一个年龄(经过了多少次GC还存活)，每经过一次GC如果该对象还没有被回收就会年龄+1。

所以为了判定哪些对象是仍然留下来的对象，JVM采用了这种类似于倒转的形式，来确定存活区哪些对象是原来有的，并且还存活下来的，从而执行年龄+1的操作

至于这个确认操作简单想一想的话应该可以按照原本空的那个区域来个遍历，然后看哪些对象在另一个存活区中，然后执行+1。但是这里是JVM，大概需要采取快速的反应方式，或许是直接将两个区取交集，然后把交集的所有对象年龄+1

老年代(Old Generation)

老年代内存空间一般情况下，里面的对象是垃圾的概率也更小。

老年代GC发生的频率比年轻代小很多。同时, 因为预期老年代中的对象大部分是存活的, 所以不再使用标记和复制(Mark and Copy)算法。而是采用移动对象的方式来实现最小化内存碎片。老年代空间的清理算法通常是建立在不同的基础上的。原则上,会执行以下这些步骤:

  1. 通过标志位(marked bit),标记所有通过 GC roots 可达的对象.
  2. 删除所有不可达对象
  3. 整理老年代空间中的内容，方法是将所有的存活对象复制,从老年代空间开始的地方,依次存放。

通过上面的描述可知, 老年代GC必须明确地进行整理,以避免内存碎片过多。

永久代(PermGen)

Java8之前有一个特殊的空间，称为“永久代”(Permanent Generation)。

它存储元数据(metadata)的地方,比如 class 信息等。此外,这个区域中也保存有其他的数据和信息, 包括内部化的字符串(internalized strings)等等。

元数据区(Metaspace)

Java 8直接删除了永久代(Permanent Generation)，改用Metaspace。将静态变量和字符串常量都放到其中。像类定义(class definitions)之类的信息会被加载到Metaspace 中。

元数据区位于本地内存(native memory)，不再影响到普通的Java对象。默认情况下, Metaspace的大小只受限于Java进程可用的本地内存。

垃圾回收原理

和我们日常的清理垃圾—->找到垃圾并且回收思路相反

JVM回收垃圾是通过标记不可达对象从而找出垃圾，也就是找到所有有用的对象，没有找到的就是垃圾

所以这里一定要区分清楚，我们这里的标记：是指标记可用对象，而不是垃圾对象。常常会有人吧这两者理解错误和混乱。

引用计数(Reference Counting)

引用计数是指从root出发，对每个引用了的对象计数+1，最后排除那些计数为0的对象或者不是从root出发的引用链。

引用计数存在的局限就是无法解决循环引用的问题。就是从非root出发的引用链。这样的引用会导致所有的对象计数器都是1，而且还没有办法追溯到这条链的源头是不是root(因为它是一个cycle)

循环引用（detached cycle）的解决方案—–弱引用之后补足

比如说可以针对于一些这种循环模式进行加入到 “弱引用”(‘weak’ references)的体系中，所以即使无法进行解决循环引用计数的场景，也可以通过弱引用实现内存回收。

Minor GC、MajorGC和FullGC执行效果

Minor GC :针对的是新生代，和存活区

Major GC ：针对的是老年区 ——-涉及到标记不可达对象，移动对象，减少碎片化内存空间

Full GC :针对的是整个堆

回到一开始的问题，如果JVM出现太多 full GC 应该怎么排查

先聊聊哪些场景会导致频繁Full GC：

  1. 内存泄漏（代码有问题，对象引用没及时释放，导致对象不能及时回收）。

  2. 死循环。

  3. 大对象。 尤其是大对象，80%以上的情况就是他。 那么大对象从哪里来的呢？

  4. 数据库（包括MySQL和MongoDB等NoSQL数据库），结果集太大。

  5. 第三方接口传输的大对象。

  6. 消息队列，消息太大。

由上面看到，大部分都是由数据库大结果集产生的。

情况1. 大数据集

大结果集可能就是因为你的 sql 查询少添加了一条 where 条件，或者 redis 中的数据没有给他设置逻辑过期时间，结果导致我们进行 dao 操作的时候一次性返回了太多的结果集，导致新创建的对象占用了太多的新生代区空间。JVM 就会频繁地GC 来使释放内存。

是否添加分页查询呢？可能有的逻辑无法添加分页查询吧

情况2. 内存泄漏

内存溢出：程序没有足够的内存使用时，就会发生内存溢出。内存溢出后程序基本上就无法正常运行了。

内存泄漏：当程序不能及时释放内存，导致占用内存逐渐增加，就是内存泄漏。内存泄漏一般不会导致程序无法运行。不过持续的内存泄漏，累积到内存上限时，就会发生内存溢出。在Java中，如果发生内存泄漏，会导致GC回收不彻底，每次GC后，堆内存使用率逐渐增高。

一般这种情况有可能就是因为 对象中出现了循环引用的情况，GC 进行回收的时候就无法对出现的对象进行回收，结果导致，堆内存使用率越来越高 ，最后出现内存溢出

这种排查就需要我们观察是否每次GC后堆内存的使用情况是否是之间升高的情况
